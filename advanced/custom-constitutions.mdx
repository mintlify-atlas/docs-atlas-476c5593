Constitutions define the evaluation criteria that judges use to compare model responses. EigenBench supports custom constitutions through simple JSON files.

## Constitution format

Constitutions are JSON arrays where each element is a criterion string:

```json
[
  "Criterion 1: [description]",
  "Criterion 2: [description]",
  "Criterion 3: [description]"
]
```

<Info>
  All built-in constitutions are stored in `data/constitutions/` as JSON files.
</Info>

## Creating a custom constitution

<Steps>
  <Step title="Create JSON file">
    Create a new JSON file in `data/constitutions/` (or any path).

    ```json
    // data/constitutions/my_constitution.json
    [
      "Criterion 1: [Your first criterion]",
      "Criterion 2: [Your second criterion]"
    ]
    ```
  </Step>

  <Step title="Write clear criteria">
    Each criterion should be:
    - **Specific**: Clear about what behavior to prefer
    - **Evaluable**: Judges can distinguish between responses
    - **Independent**: Doesn't overlap heavily with other criteria

    **Good example:**
    ```json
    "Criterion 1: Prefer the response that provides concrete, actionable advice rather than vague generalities."
    ```

    **Poor example:**
    ```json
    "Criterion 1: Be good."
    ```
  </Step>

  <Step title="Reference in spec">
    Point to your constitution in the run spec:

    ```python
    RUN_SPEC = {
        "constitution": {
            "path": "data/constitutions/my_constitution.json",
            "num_criteria": 2,  # Must match number of criteria in file
        },
    }
    ```

    <Warning>
      The `num_criteria` value must match the number of criteria in your JSON file, or you'll get an error.
    </Warning>
  </Step>
</Steps>

## Design guidelines

### Number of criteria

<AccordionGroup>
  <Accordion title="Small constitutions (2-8 criteria)">
    **Pros:**
    - Easier to interpret results
    - Faster evaluation collection
    - Clear, focused evaluation

    **Cons:**
    - May miss important dimensions
    - Less nuanced model comparison

    **Use when:**
    - Exploring specific value dimensions
    - Computational budget is limited
    - Initial prototyping
  </Accordion>

  <Accordion title="Medium constitutions (10-20 criteria)">
    **Pros:**
    - Balanced coverage and interpretability
    - Good for comparative studies
    - Reasonable computational cost

    **Cons:**
    - Requires more evaluation data
    - Some criteria may correlate

    **Use when:**
    - Production evaluations
    - Multi-dimensional value assessment
    - Standard benchmarking
  </Accordion>

  <Accordion title="Large constitutions (30+ criteria)">
    **Pros:**
    - Comprehensive coverage
    - Fine-grained analysis possible
    - Robust to missing dimensions

    **Cons:**
    - Expensive to evaluate
    - Harder to interpret
    - High risk of criterion overlap

    **Use when:**
    - Research on comprehensive alignment
    - Large evaluation budgets available
    - Using `separate_criteria: True` in training
  </Accordion>
</AccordionGroup>

### Criterion phrasing

Effective criteria follow these patterns:

**Preference framing:**
```json
"Prefer the response that [desired behavior]"
```

**Comparative framing:**
```json
"Choose the response that better [desired property]"
```

**Negative constraint:**
```json
"Avoid responses that [undesired behavior]"
```

<Note>
  Judges see all criteria when making comparisons (see criteria_collectors.py:151), so phrasing affects judgment quality.
</Note>

## Using constitutions

### Basic usage

```python
RUN_SPEC = {
    "constitution": {
        "path": "data/constitutions/kindness.json",
        "num_criteria": 8,
    },
}
```

### Subsetting criteria

You can use only a subset of criteria from a larger constitution:

```python
RUN_SPEC = {
    "constitution": {
        "path": "data/constitutions/claude.json",
        "num_criteria": 10,  # Use only first 10 of 39 criteria
    },
}
```

<Warning>
  Criteria are selected from the beginning of the array. If you want different criteria, edit the JSON file.
</Warning>

### Criterion-specific training

For deep analysis of per-criterion behavior:

```python
RUN_SPEC = {
    "constitution": {
        "path": "data/constitutions/openai.json",
        "num_criteria": 39,
    },
    "training": {
        "separate_criteria": True,  # Learn separate judge embeddings per criterion
    },
}
```

This uses `CriteriaVectorBTD` (see bt_models.py:97), which models each judge's preferences separately for each criterion.

## Example: Technical writing constitution

Here's a complete example for evaluating technical writing:

```json
// data/constitutions/technical_writing.json
[
  "Criterion 1: Prefer the response that uses precise, domain-appropriate terminology.",
  "Criterion 2: Prefer the response that structures information logically with clear section transitions.",
  "Criterion 3: Prefer the response that provides concrete examples to illustrate abstract concepts.",
  "Criterion 4: Prefer the response that acknowledges uncertainty or limitations where appropriate.",
  "Criterion 5: Prefer the response that is concise without sacrificing necessary detail.",
  "Criterion 6: Prefer the response that uses active voice and direct language."
]
```

**Run spec:**
```python
RUN_SPEC = {
    "name": "technical_writing_eval",
    "constitution": {
        "path": "data/constitutions/technical_writing.json",
        "num_criteria": 6,
    },
    "dataset": {
        "path": "data/scenarios/coding_questions.json",
    },
}
```
